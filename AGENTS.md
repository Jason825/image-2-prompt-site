# 项目级 AGENTS.md

## 项目背景

这个项目是一个轻量的内容网站，聚焦于整理高质量的 Image 2.0 提示词和对应案例图。

项目当前阶段的目标，是帮助用户快速浏览、搜索、理解、复制和复用提示词，并通过清晰、简洁的内容结构完成产品验证。

当前它不是社区产品，不是完整 SaaS，也不是复杂创作者平台。

## 当前产品方向

- 优先做好“提示词发现与浏览体验”
- 保持网站轻量、清晰、好理解
- 更看重快速搜索、便捷浏览、直接复用，而不是功能堆叠
- 整体应该更像“精选提示词内容站”，而不是后台系统或复杂平台

## 当前信息架构

对外公开的信息结构已经刻意简化。

当前主要公开页面：

- `/explore`：浏览与搜索
- `/categories`：按内容类型浏览
- `/collections`：按专题集合浏览
- `/updates`：项目更新记录
- `/prompts/[slug]`：提示词详情页

辅助页面：

- `/sources`：内容来源说明

除非用户明确要求，否则不要重新增加不必要的公开页面入口。

## 默认不做的功能

除非用户明确提出，否则不要主动加入以下功能：

- 用户登录
- 会员系统
- 支付系统
- 评论或社区功能
- 公开标签导航
- 收藏或书签系统
- 后台管理系统
- CMS 接入
- 站内 Prompt 改写/生成工具

## 内容结构约束

- Prompt 是核心内容单元
- 分类和专题是当前主要组织方式
- 标签可以保留在数据层，供内部筛选或未来扩展使用，但默认不要作为对外主结构
- Prompt 卡片和详情页应优先服务于“理解、复制、复用”

## UI 和 UX 约束

- 默认使用中文界面文案
- 风格保持简洁、现代、商业化、可信
- 避免页面拥挤、入口重复、筛选层级过多
- 优先更少但更明确的操作
- 搜索应该是全站主动作之一
- 页面要让非技术用户也能快速理解

## 工程约束

- 改动尽量聚焦、克制
- 优先复用现有结构和组件
- 不要随意引入大型依赖，除非确实有明显收益
- 优先选择容易上线、容易维护的方案
- 不要主动重做项目架构，除非用户明确要求
- 如果需要提交 Git，默认使用中文提交信息，除非用户明确要求使用其他语言

## 统计与排序约束

- 如果需要“热门”“排序”“推荐”等能力，优先采用轻量、可落地的统计方案
- 优先使用简单事件计数，而不是复杂分析系统
- 先服务于产品验证，不要为了统计系统本身过度设计

## 验证要求

在修改 Web 项目后，只要条件允许，优先运行这些检查：

- `npm run lint`
- `npm run build`

如果改动影响运行逻辑，需要说明本地如何验证。

## 输出偏好

完成任务后，输出尽量简洁，并优先说明：

1. 做了什么
2. 关键改了哪些文件
3. 如何运行或验证
4. 有哪些风险、限制或下一步建议

## PRD 文档规则

- 以后在这个项目里写或修改 `PRD` 文档时，必须保持文档干净、可执行、可直接阅读
- `PRD` 里只保留当前最终采用的方案，不要写已经放弃的方案
- 不要把“讨论过程”“尝试过但不用的路线”“历史岔路”混进正式 `PRD`
- 如果方案发生变化，应直接把 `PRD` 更新为最新结论，而不是把旧方案和新方案同时保留在文档里
