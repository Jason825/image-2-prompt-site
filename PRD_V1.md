# ImagePromptive PRD V1

## 1. 项目目标

`ImagePromptive` 是一个围绕 `Image 2.0` 提示词的轻量内容站。

当前阶段的目标不是做复杂社区、会员系统或 SaaS，而是先把下面这几件事做扎实：

- 让用户快速找到可直接复用的高质量提示词
- 让用户方便地浏览、搜索、查看案例图、复制提示词
- 通过轻量统计能力识别哪些作品更热门
- 让整个项目可以稳定上线、持续更新和低成本维护

## 2. 产品定位

一句话定位：

`一个专注于 Image 2.0 提示词浏览、搜索、案例展示与热门内容识别的轻量内容站。`

当前不把项目定位为：

- 社区产品
- 会员平台
- 在线出图工具
- 复杂的后台系统

## 3. 当前功能范围

### 3.1 对外页面

当前保留的核心页面：

- `/`：首页
- `/explore`：浏览与搜索
- `/categories`：分类浏览
- `/collections`：专题浏览
- `/prompts/[slug]`：提示词详情页
- `/sources`：来源说明页

当前不保留或不开放的内容：

- `更新` 入口暂时隐藏
- `标签` 不作为公开主导航
- `收藏` 暂不实现
- `关于` 页面不做

### 3.2 V1 必做功能

- 首页展示站点价值、精选内容、分类入口、专题入口
- 全站统一搜索入口
- 提示词列表浏览与排序
- 分类浏览
- 专题浏览
- 提示词详情页
- 一键复制提示词
- 下载示例图
- 基于真实数据的热门排序

### 3.3 当前不做

- 用户登录
- 收藏系统
- 评论与社区
- 支付系统
- 后台 CMS
- 复杂推荐系统
- 站内 Prompt 改写工具

## 4. 用户核心任务

用户来到网站后，主要做这几件事：

1. 浏览灵感
2. 搜索合适的提示词
3. 查看案例图和说明
4. 复制提示词去使用

站点运营侧还需要做到：

5. 知道哪些内容更热门，便于调整首页和专题

## 5. 技术方案

### 5.1 当前架构

当前采用一体化方案，不做前后端分离：

- 前端页面：`Next.js App Router`
- 接口层：`Next.js Route Handlers`
- 数据库：`Postgres`
- 部署环境：`腾讯云轻量应用服务器`
- 反向代理：`Nginx`
- 容器编排：`Docker Compose`

### 5.2 为什么采用这套方案

- 对当前阶段来说实现成本低
- 更适合国内访问
- 不依赖海外平台注册与部署链路
- 便于持久化统计数据
- 后续可以继续扩展，而不是一次性过度设计

## 6. 统计目标

V1 只统计最关键的三类行为：

- `view`：进入提示词详情页
- `copy`：点击复制提示词
- `download`：点击下载示例图

## 7. 热门排序规则

当前使用简单可解释的评分公式：

`score = views * 1 + downloads * 2 + copies * 3`

说明：

- 浏览代表兴趣
- 下载代表更强意图
- 复制代表最高复用意图

## 8. 数据库设计

### 8.1 prompt_stats

当前只保留一张核心统计表：

- `slug`
- `views`
- `copies`
- `downloads`
- `score`
- `updated_at`

当前不做事件明细表，不做复杂埋点平台。

## 9. 接口设计

### 9.1 POST /api/prompt-stats

用途：

- 上报提示词行为事件

请求字段：

- `slug`
- `eventType`

### 9.2 GET /api/prompt-stats

用途：

- 返回当前提示词统计数据
- 支持前端“最热门”排序

## 10. 前端接入方式

### 10.1 浏览统计

用户进入 `/prompts/[slug]` 时触发一次 `view` 上报。

当前已改为带冷却时间的浏览计数，不再是同一会话永久只记一次。

### 10.2 复制统计

用户点击“复制提示词”时触发 `copy` 上报。

### 10.3 下载统计

用户点击下载图片时触发 `download` 上报。

### 10.4 热门排序

首页和浏览页都应基于真实 `score` 排序，而不是仅依赖手动写死的 `popularityScore`。

## 11. 列表展示规则

站点保留瀑布流视觉风格，但排序必须满足：

- 热门顺序按阅读顺序体现
- 从左到右、从上到下展示
- 不能因为假瀑布流实现导致“分数最高却看起来不在第一”

因此当前应使用真正保持阅读顺序的 Masonry 实现，而不是单纯依赖 CSS `column-count` 的伪瀑布流。

## 12. 部署方案

### 12.1 运行环境

当前服务器方案：

- `腾讯云轻量应用服务器`
- `Ubuntu 22.04`
- `Docker / Docker Compose`

### 12.2 手动部署

服务器部署的基础方式仍然是：

- 拉取代码
- 执行 `docker compose up -d --build`

### 12.3 自动部署最终选型

当前最终只保留这一套自动部署方案：

- `GitHub Actions`
- `SSH` 登录腾讯云服务器
- 服务器执行 `deploy/deploy.sh`
- `Docker Compose` 自动重建并上线

当前最终发布链路：

`本地 push -> GitHub Actions -> SSH 登录腾讯云服务器 -> 执行 deploy.sh -> docker compose up -d --build`

## 13. 自动部署所需内容

### 13.1 仓库内保留的文件

- `.github/workflows/deploy.yml`
- `deploy/deploy.sh`

### 13.2 GitHub Secrets

仓库中需要配置：

- `DEPLOY_HOST`
- `DEPLOY_PORT`
- `DEPLOY_USER`
- `DEPLOY_PATH`
- `DEPLOY_SSH_KEY`

### 13.3 服务器要求

服务器需要满足：

- 已配置 SSH 公钥登录
- 项目目录存在于 `/www/image-2-prompt-site`
- `deploy/deploy.sh` 具备执行权限
- Docker 与 Docker Compose 可正常运行

## 14. 风险与限制

### 14.1 当前方案的优点

- 可上线
- 可持续更新内容
- 可识别热门内容
- 成本可控
- 运维复杂度相对较低

### 14.2 当前方案的限制

- 仍不是完整后台系统
- 统计是轻量事件计数，不是专业分析平台
- 单机部署在流量上来后仍需进一步升级

## 15. 当前实施原则

这一版只做最小可用闭环：

- 真实统计
- 热门排序
- 可上线部署
- 自动部署打通

不做：

- 登录
- 收藏
- 复杂后台
- 多套部署方案并存
- 为未来假设需求做过度设计

## 16. 成功标准

如果 V1 达到下面这些状态，就可以认为方向成立：

- 用户能快速找到提示词
- 用户愿意频繁点击复制
- 站点可以基于真实数据识别热门内容
- 首页与浏览页能真实反映热门排序
- 发布流程从“手工登录服务器”收敛为“推代码后自动部署”
