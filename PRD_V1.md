# ImagePromptive PRD V1

## 1. 项目目标

本项目是一个面向 `Image 2.0` 场景的提示词内容站，核心目标不是做复杂社区或 SaaS，而是先把下面这件事做扎实：

- 让用户快速找到可直接复用的高质量提示词
- 让用户可以方便地浏览、搜索、复制、查看示例效果
- 通过轻量统计能力，识别哪些作品更受欢迎

当前阶段优先做出一个可上线、可验证、可持续补内容的版本。

## 2. 产品定位

一句话定位：

`一个专注于 Image 2.0 提示词浏览、搜索与案例展示的轻量内容站。`

当前不把项目定位为：

- 社区产品
- 登录会员平台
- 在线出图工具
- 复杂后台系统

## 3. 当前功能范围

### 3.1 对外页面

当前站点的公开结构以精简为原则，保留以下核心页面：

- `/`：首页
- `/explore`：浏览与搜索
- `/categories`：分类浏览
- `/collections`：专题推荐
- `/prompts/[slug]`：提示词详情页
- `/sources`：来源说明页

说明：

- `更新` 入口当前隐藏
- `标签` 不作为对外主导航
- `收藏` 暂不做
- `关于` 页面不做

### 3.2 V1 必做功能

- 首页展示站点价值、精选内容、分类入口、专题推荐
- 全站统一搜索入口
- 提示词列表浏览与基础排序
- 分类浏览
- 专题浏览
- 提示词详情页
- 一键复制提示词
- 基于真实数据的热门排序

### 3.3 当前不做

- 用户登录
- 收藏系统
- 评论与社区
- 支付系统
- 后台 CMS
- 站内 AI 改写 Prompt
- 复杂推荐系统

## 4. 用户核心任务

用户来到站点后，主要做这几件事：

1. 浏览灵感
2. 搜索合适的提示词
3. 查看案例图和说明
4. 复制提示词去使用

后续站点运营方还需要一件事：

5. 知道哪些内容更热门，便于调整首页、专题和内容策略

## 5. 为什么采用腾讯云轻量服务器方案

当前阶段推荐组合：

- 前端与接口：`Next.js`
- 部署服务器：`腾讯云轻量应用服务器`
- 数据库：`Postgres`
- 反向代理：`Nginx`

选择这套组合的原因：

- 不依赖国外平台账号体系和注册流程
- 国内访问更稳定
- 成本低，适合个人项目和 MVP
- 能持久化真实统计数据，不再依赖手动 `popularityScore`
- 后续如果要加后台、登录、内容管理，也更容易继续扩展

## 6. 技术方案

### 6.1 架构方式

当前不做前后端分离，采用一体化方案：

- 前端页面继续用 `Next.js App Router`
- 后端接口使用 `Next.js Route Handlers`
- 应用部署在 `腾讯云轻量应用服务器`
- 数据库存放在同一台服务器上的 `Postgres`
- 使用 `Nginx` 负责反向代理和后续 HTTPS

这样做的优点：

- 一个仓库即可完成页面、接口和部署
- 成本低，维护简单
- 不需要额外购买独立数据库服务
- 更适合当前内容站形态

### 6.2 为什么不继续纯前端手动热门

纯前端方案虽然能快速上线，但有明显限制：

- 无法知道真实浏览量
- 无法知道复制量
- 无法知道下载量
- “最热门”只能靠手动维护分数

因此如果要真正做“最热门”，就需要最小后端能力和数据库支持。

### 6.3 为什么不继续使用 Vercel + Supabase

这套方案并不是技术上不能做，而是当前阶段不适合本项目：

- `Vercel` 注册与验证流程对当前使用条件不友好
- 海外平台对国内访问的稳定性和速度存在不确定性
- `Supabase` 也是海外服务，整体链路不够本地化
- 对早期项目来说，单机服务器部署更便宜、更直接、更可控

因此当前阶段优先选择：

`腾讯云轻量服务器 + Next.js + Postgres + Nginx`

## 7. 第一阶段统计目标

V1 只统计最关键的 3 类行为：

- `view`：进入提示词详情页
- `copy`：点击复制提示词
- `download`：点击下载示例图

这三类数据已经足够支撑热门排序，不需要一开始就做复杂埋点系统。

## 8. 热门排序规则

V1 采用简单、可解释的评分公式：

`score = views * 1 + downloads * 2 + copies * 3`

解释：

- 浏览代表兴趣
- 下载代表更强意图
- 复制代表最高复用意图

这套权重先用于 MVP，后续可以根据实际数据再调整。

## 9. 数据库设计

### 9.1 表一：prompt_stats

用于存储每条提示词的聚合统计结果。

建议字段：

- `slug`：提示词唯一标识
- `views`：浏览次数
- `copies`：复制次数
- `downloads`：下载次数
- `score`：热门分数
- `updated_at`：最后更新时间

说明：

- 第一版先做聚合表，不急着做明细事件表
- 如果后续要看更细的行为轨迹，再补事件日志表

### 9.2 可选表二：prompt_events

这张表不是 V1 必做，但可以作为后续扩展预留。

可能字段：

- `id`
- `slug`
- `event_type`
- `created_at`
- `ip_hash`
- `user_agent`

V1 暂时不建议一开始就上线这张表，先保持轻量。

## 10. 接口设计

### 10.1 上报事件

`POST /api/prompt-stats`

用途：

- 前端发生关键行为时调用
- 上报 `slug` 和 `eventType`
- 服务端更新对应提示词的聚合统计

请求示例：

```json
{
  "slug": "minimal-product-photo",
  "eventType": "copy"
}
```

### 10.2 获取统计结果

`GET /api/prompt-stats`

用途：

- 返回当前提示词统计数据
- 支持前端“最热门”排序
- 后续也可给简单后台使用

返回字段至少包含：

- `slug`
- `views`
- `copies`
- `downloads`
- `score`

## 11. 前端接入方式

### 11.1 详情页浏览统计

用户进入 `/prompts/[slug]` 时，前端触发一次 `view` 上报。

### 11.2 复制统计

用户点击“复制提示词”按钮时，触发一次 `copy` 上报。

### 11.3 下载统计

用户点击下载示例图时，触发一次 `download` 上报。

### 11.4 浏览页热门排序

`/explore` 页的“最热门”排序不再使用手动分数，而是：

- 请求统计接口
- 获取每条提示词的 `score`
- 按 `score` 从高到低排序

## 12. 部署方案

### 12.1 服务器

优先选择：`腾讯云轻量应用服务器`

建议初始配置：

- `2核 2G` 即可起步
- `Ubuntu 22.04`
- 先以单机部署为主

原因：

- 成本最低
- 够支撑早期流量
- 易于维护
- 后续可以平滑升级

### 12.2 运行方式

推荐部署结构：

- `Next.js` 应用进程
- `Postgres` 数据库
- `Nginx` 反向代理

推荐方式：

- 优先使用 `Docker Compose`
- 所有服务放在同一台服务器内

这样做的好处：

- 部署简单
- 迁移方便
- 环境一致性更高

### 12.3 环境变量

预计需要：

- `DATABASE_URL`
- `NODE_ENV`
- `PORT`

后续如果接入对象存储、邮件或后台能力，再补充新的环境变量。

## 13. 风险与限制

### 13.1 当前方案的优势

- 可部署
- 可持续更新内容
- 可拿到真实热门数据
- 国内访问更稳定
- 成本可控

### 13.2 当前方案的限制

- 仍然不是完整后台系统
- 统计是轻量事件计数，不是专业分析平台
- 单机部署在流量上来后需要考虑拆分

### 13.3 当前阶段可接受的取舍

当前阶段重点是产品验证，不追求：

- 高并发架构
- 多机高可用
- 完整运维体系
- 复杂数据分析看板

## 14. 当前实施原则

这一版只做最小可用闭环：

- 真实统计
- 热门排序
- 不做复杂后台
- 不做登录
- 不做收藏
- 不做大规模架构重构

原则是：

- 先让“最热门”真正跑通
- 先让内容站具备基本数据反馈能力
- 先服务产品验证，再考虑系统扩张

## 15. 成功标准

如果 V1 达到下面这些状态，就可以认为方向成立：

- 用户能快速找到提示词
- 用户愿意频繁点击复制
- 站点可以基于真实数据识别热门内容
- 运营上可以据此调整首页、专题和内容优先级
- 技术架构足够轻，便于继续上线和维护

## 16. 下一步开发范围

确认本方案后，下一步开发按以下顺序推进：

1. 把统计存储从海外服务切到自建 `Postgres`
2. 建立 `prompt_stats` 表
3. 保留现有统计 API，改为读写自建数据库
4. 完成服务器部署脚本或 `Docker Compose`
5. 接入 `Nginx`
6. 完成首轮国内服务器部署
## 17. 自动部署方案

### 17.1 当前为什么需要自动部署

当前手动部署流程通常是：

1. 本地修改代码
2. 提交到 GitHub
3. 等待 Gitee 同步
4. 登录腾讯云服务器
5. 手动执行 `git pull`
6. 手动执行 `docker compose up -d --build`

这套流程在早期可以接受，但随着功能增多，会逐渐出现以下问题：

- 发布步骤多，容易漏操作
- 本地版本、GitHub、Gitee、服务器版本不一致时排查困难
- 每次小改动都要重复登录服务器
- 回滚成本高
- 一天多次发版时效率低

因此，当前阶段建议引入轻量自动部署，而不是继续长期依赖纯手工更新。

### 17.2 当前阶段最适合的自动部署架构

在不引入重型 CI/CD 平台的前提下，当前推荐方案为：

- GitHub：主开发仓库
- Gitee：国内镜像仓库与部署触发源
- 腾讯云轻量应用服务器：线上运行环境
- Gitee WebHook：代码同步后的自动触发器
- 服务器部署脚本：真正执行拉代码与重启容器
- 部署监听服务：接收 WebHook 请求并调用部署脚本
- Nginx：将外部 WebHook 请求转发到部署监听服务

推荐链路如下：

`本地 push 到 GitHub -> Gitee 自动同步 -> Gitee Push WebHook -> 腾讯云服务器收到请求 -> 自动执行 deploy.sh -> Docker Compose 重建并上线`

### 17.3 为什么不直接上重型 CI/CD

当前项目仍处于产品验证阶段，不建议一开始就引入 Jenkins、自建 GitHub Actions Runner、复杂发布流水线等重型方案，原因包括：

- 配置与维护成本高
- 故障排查链路更长
- 对当前站点规模来说收益不够大
- 会拖慢功能验证节奏

因此当前阶段更适合：

- 保留简单部署结构
- 只把“登录服务器手动发版”这一段自动化
- 等项目进入稳定运营阶段后，再考虑完整 CI/CD

### 17.4 自动部署的服务端组成

服务器上需要增加以下最小组件：

#### A. 部署脚本 `deploy.sh`

职责：

- 进入项目目录
- 从 Gitee 拉取最新 `main`
- 执行 `docker compose up -d --build`
- 记录部署日志
- 输出当前容器状态

目标是让服务器真正的部署动作被封装成一个脚本，而不是散落成多条手工命令。

#### B. 部署监听服务

职责：

- 监听来自 Gitee 的 WebHook 请求
- 校验 WebHook 密钥
- 判断是否为 `main` 分支的 push
- 调用 `deploy.sh`
- 返回部署触发结果

该服务应保持极简，只负责触发，不承载业务逻辑。

#### C. Nginx 路由

当前站点已经使用 Nginx，因此自动部署接口建议通过 Nginx 暴露，例如：

- `https://imagepromptive.top/deploy-hook`

由 Nginx 再转发到本机监听服务，例如：

- `127.0.0.1:9001`

这样做的好处是：

- 不需要额外暴露新的公网端口
- 更容易统一 HTTPS
- 更容易做访问限制与日志记录

### 17.5 自动部署触发规则

为保证稳定性，当前建议规则如下：

- 只接受 `POST` 请求
- 只处理 `main` 分支的 push
- 必须校验 WebHook 密钥
- 只允许 Gitee 同步后的推送触发部署
- 一次只运行一个部署任务，避免重复触发导致并发部署

### 17.6 自动部署阶段的发布方式

引入自动部署后，推荐发布方式变为：

1. 本地开发并测试
2. 提交代码
3. push 到 GitHub
4. 等待 Gitee 自动同步
5. Gitee 自动通知服务器
6. 服务器自动完成拉取与重建

也就是说，发布动作会从“手工登录服务器部署”收敛为“提交并推送代码”。

### 17.7 安全要求

自动部署虽然简化了流程，但必须补充基础安全措施：

- WebHook 必须使用独立密钥
- 部署接口不能无鉴权公开调用
- 部署服务只监听本机地址，由 Nginx 转发
- 部署日志需要落盘，便于排查问题
- 部署脚本必须固定仓库目录，不能接受任意路径参数
- 部署脚本执行用户应尽量受限，不直接暴露额外系统权限

### 17.8 当前阶段的取舍

当前阶段不追求：

- 多环境复杂发布系统
- 灰度发布
- 蓝绿部署
- 自动回滚系统
- 多服务器集群编排

当前阶段只追求：

- 推代码后自动更新服务器
- 保持部署链路尽量短
- 出问题时容易定位
- 成本尽量低

### 17.9 后续可升级方向

如果后续项目持续运营，并且发布频率、流量、可用性要求提升，可以逐步升级为：

- 增加测试后再部署
- 增加部署失败告警
- 增加镜像版本管理
- 增加回滚脚本
- 增加独立后台与多服务拆分
- 增加更完整的 CI/CD 流水线

但在当前阶段，这些都不作为必须项。
