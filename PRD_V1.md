# ImagePromptive PRD V1

## 1. 项目目标

`ImagePromptive` 是一个围绕 `Image 2.0` 提示词的轻量内容站。

当前阶段的目标不是做复杂社区、会员系统或 SaaS，而是先把下面几件事做扎实：

- 让用户快速找到可直接复用的高质量提示词
- 让用户方便地浏览、搜索、查看案例图、复制提示词
- 通过轻量统计识别哪些内容更热门
- 通过一个低成本付费精选包验证用户是否愿意为“节省试错时间”付费
- 让整站可以稳定上线、持续更新、低成本维护

## 2. 产品定位

一句话定位：

`一个专注于 Image 2.0 提示词浏览、搜索、案例展示、热门内容识别和轻量付费验证的精选内容站。`

## 3. 当前功能范围

### 3.1 公开页面

- `/`：首页
- `/explore`：浏览与搜索
- `/categories`：按内容类型浏览
- `/collections`：按专题浏览
- `/pro`：付费精选包介绍页
- `/prompts/[slug]`：提示词详情页
- `/sources`：内容来源说明

### 3.2 V1 必做功能

- 首页展示精选内容、热门内容、分类入口和付费精选包入口
- 全站统一搜索入口
- 提示词列表浏览与排序
- 分类浏览
- 专题浏览
- 提示词详情页
- 一键复制提示词
- 下载示例图
- 基于真实统计数据的热门排序
- 付费精选包介绍页和购买意向入口

### 3.3 当前不做

- 用户登录
- 收藏系统
- 评论与社区
- 在线支付系统
- 后台 CMS
- 复杂推荐系统
- 站内 Prompt 改写工具
- 自动发货系统

## 4. 用户核心任务

用户来到网站后，主要做这几件事：

1. 浏览灵感
2. 搜索合适的提示词
3. 查看案例图和说明
4. 复制提示词去使用
5. 如果觉得有价值，了解并联系购买精选包

站点运营侧需要做到：

1. 知道哪些内容更热门
2. 知道用户是否愿意为整理好的提示词包付费
3. 根据真实反馈继续补充内容和优化转化路径

## 5. 付费 MVP 方案

### 5.1 当前采用方案

先做一个低成本的付费提示词精选包：

- 产品名称：`Image 2.0 商业提示词精选包`
- 首发验证价：`¥29`
- 原价展示：`¥59`
- 交付方式：早期人工确认和发送文件
- 购买入口：邮箱联系 `763484227@qq.com`

### 5.2 为什么先这样做

当前最重要的是验证用户是否愿意付费，而不是马上搭建复杂支付、登录和自动交付系统。

先用一个简单的购买意向入口，可以快速验证：

- 用户是否点击付费页
- 用户是否愿意发邮件咨询
- 哪些提示词场景最容易促成购买
- 定价是否有调整空间

### 5.3 精选包内容方向

精选包优先覆盖高商业复用场景：

- 电商图
- 品牌海报
- 社媒封面
- UI 截图
- 写实摄影
- 信息图

## 6. 技术方案

### 6.1 当前架构

- 前端页面：`Next.js App Router`
- 接口层：`Next.js Route Handlers`
- 数据库：`Postgres`
- 部署环境：`腾讯云轻量应用服务器`
- 反向代理：`Nginx`
- 容器编排：`Docker Compose`

### 6.2 为什么采用这套方案

- 当前阶段实现成本低
- 适合国内访问
- 方便持久化统计数据
- 便于后续继续扩展

## 7. 统计目标

V1 只统计最关键的三类行为：

- `view`：进入提示词详情页
- `copy`：点击复制提示词
- `download`：点击下载示例图

## 8. 热门排序规则

当前评分公式：

```txt
score = views * 1 + downloads * 2 + copies * 3
```

说明：

- 浏览代表兴趣
- 下载代表更强意图
- 复制代表最高复用意图

## 9. 数据库设计

### 9.1 prompt_stats

当前只保留一张核心统计表：

- `slug`
- `views`
- `copies`
- `downloads`
- `score`
- `updated_at`

## 10. 接口设计

### 10.1 POST /api/prompt-stats

用途：

- 上报提示词行为事件

请求字段：

- `slug`
- `eventType`

### 10.2 GET /api/prompt-stats

用途：

- 返回当前提示词统计数据
- 支持前端热门排序

## 11. 前端接入方式

### 11.1 浏览统计

用户进入 `/prompts/[slug]` 时触发 `view` 上报。

### 11.2 复制统计

用户点击“复制提示词”时触发 `copy` 上报。

### 11.3 下载统计

用户点击下载图片时触发 `download` 上报。

### 11.4 防刷规则

当前先采用前端轻量冷却策略：

- 同一设备、同一作品 `view`：`1 分钟` 内只记一次
- 同一设备、同一作品 `copy`：`10 分钟` 内只记一次
- 同一设备、同一作品 `download`：`10 分钟` 内只记一次

### 11.5 热门排序

首页和浏览页都应优先基于真实 `score` 排序。

当后端统计可用时，不再混用静态 `popularityScore`。

## 12. 列表展示规则

站点保留瀑布流视觉风格，但排序必须满足：

- 热门顺序按阅读顺序体现
- 从左到右、从上到下展示
- 不能出现“分数最高却看起来不在第一”的情况

因此当前使用真正保持阅读顺序的 Masonry 实现，而不是单纯依赖 CSS `column-count` 的伪瀑布流。

## 13. 部署方案

### 13.1 运行环境

- `腾讯云轻量应用服务器`
- `Ubuntu 22.04`
- `Docker / Docker Compose`

### 13.2 自动部署最终方案

当前最终只保留这一套自动部署链路：

```txt
本地推送到 Gitee -> Gitee WebHook -> 腾讯云服务器 -> 执行 deploy.sh -> docker compose up -d --build
```

### 13.3 代码仓库策略

- `Gitee`：主部署源
- `GitHub`：备份与公开展示仓库

日常发布以 Gitee 为准。

## 14. 自动部署所需内容

### 14.1 仓库内文件

- `deploy/deploy.sh`
- `deploy/hook_server.py`
- `deploy/hook.env.example`
- `deploy/imagepromptive-deploy-hook.service`
- `deploy/push-all.ps1`

### 14.2 服务器运行要求

服务器需要满足：

- 项目目录存在于 `/www/image-2-prompt-site`
- `deploy/deploy.sh` 具备执行权限
- Python 3 可运行 `hook_server.py`
- `docker compose` 可被部署用户通过 `sudo` 执行
- systemd 已启用部署监听服务
- 部署监听服务可以被 Nginx 容器反向代理访问

### 14.3 Gitee WebHook 要求

Gitee 仓库需要配置：

- WebHook 地址：`https://imagepromptive.top/deploy-hook`
- 触发事件：`Push`
- 密钥：与服务器 `deploy/hook.env` 中的 `HOOK_SECRET` 一致

## 16. 风险与限制

### 16.1 当前方案优点

- 国内链路更稳
- 自动部署路径短
- 可持续更新
- 成本可控
- 付费验证成本低

### 16.2 当前方案限制

- 仍然是单机部署
- 统计是轻量事件计数，不是专业分析平台
- 付费精选包当前为人工交付，不是自动交易系统
- 如果以后流量或订单明显增长，需要升级部署、统计、支付和交付方案

## 17. 成功标准

如果 V1 达到下面这些状态，就可以认为方向成立：

- 用户能快速找到提示词
- 用户愿意频繁点击复制
- 站点可以基于真实数据识别热门内容
- 首页与浏览页可以真实反映热门排序
- 发布流程从手动登录服务器收敛为推送到 Gitee 后自动部署
- 有用户点击或主动联系购买精选包
