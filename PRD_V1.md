# ImagePromptive PRD V1

## 1. 文档目标

这份文档用于明确 `Image 2.0` 提示词网站第一阶段产品的目标、范围、核心页面、关键功能和交付边界，作为后续设计、开发和内容准备的执行依据。

## 2. 产品背景

当前 AI 图片生成需求越来越强，但多数用户在实际使用中会遇到几个问题：

- 不知道该怎么写 prompt
- 能搜到很多 prompt，但难以判断哪些真正可用
- 缺少结构化整理，复制之后仍然不会改
- 不同场景缺少现成模板，生成效率低

因此，本项目希望做一个只服务 `Image 2.0` 的垂直提示词网站，用结构化案例帮助用户：

- 快速找到可用 prompt
- 直接复制复用
- 学会如何改写与扩展

## 3. 产品目标

### 3.1 第一阶段目标

在第一阶段，网站需要完成以下目标：

- 建立一个可浏览、可搜索、可筛选的 `Image 2.0` Prompt 内容库
- 让用户能快速查看效果图并复制 Prompt
- 建立统一的数据结构，为后续扩展工具功能打基础
- 通过高质量精选内容验证用户是否愿意持续使用

### 3.2 不在第一阶段解决的问题

第一阶段不追求：

- 站内完整出图
- 完整用户社区
- 团队协作
- 企业功能
- API 能力

## 4. 产品定位

### 一句话定位

一个专注于 `Image 2.0` 的提示词库与案例网站，帮助用户发现、复制、理解和优化高质量 Prompt。

### 产品形态

第一阶段产品形态为：

`提示词库 + 效果展示`

第二阶段再逐步加入：

`轻工具能力`

## 5. 目标用户

### 5.1 新手用户

特征：

- 刚开始使用 AI 图片生成
- 不熟悉 Prompt 写法
- 希望直接复制现成内容

核心诉求：

- 找到能直接用的 Prompt
- 看图判断效果

### 5.2 设计师 / 内容创作者

特征：

- 需要持续寻找灵感
- 对视觉风格和出图效率较敏感

核心诉求：

- 快速筛选高质量案例
- 找到适合某种风格或场景的 Prompt
- 对已有 Prompt 做二次改写

### 5.3 电商 / 营销用户

特征：

- 更看重出图效率和模板复用
- 需要大量产品图、海报图、社媒图

核心诉求：

- 按场景快速找到模板
- 直接复制并替换主体内容

## 6. 用户核心任务

用户来到网站后，核心任务主要有 4 类：

1. 浏览灵感
2. 搜索合适 Prompt
3. 复制 Prompt 去使用
4. 基于已有 Prompt 继续修改

第一阶段优先解决前 3 类任务。

## 7. 第一阶段功能范围

## 7.1 必做功能

首页目标：

- 让用户快速理解网站是做什么的

- 展示精选内容和热门分类

首页模块建议：

- 顶部主标题与价值描述

- 热门分类入口

- 专题入口

### 7.1.2 Prompt 列表页

页面目标：

- 支持按条件快速浏览 Prompt
- 帮助用户找到最适合当前场景的内容

功能要求：

- 卡片列表展示
- 展示缩略图、标题、标签、简短说明
- 支持按关键词搜索
- 支持按分类筛选
- 支持按标签筛选
- 支持按排序浏览

### 7.1.3 Prompt 详情页

页面目标：

- 让用户完整查看一个 Prompt 的结构化内容
- 支持一键复制

核心模块：

- 标题
- 效果图
- 主 Prompt
- Negative Prompt
- 参数信息
- 标签与分类
- 使用场景说明
- 补充备注
- 一键复制按钮
- 相关推荐

### 7.1.4 搜索与筛选

搜索目标：

- 允许用户用少量关键词快速找到可用内容

筛选维度建议：

- 分类
- 风格
- 用途
- 是否人物
- 是否产品图
- 难度等级

### 7.1.5 一键复制

功能目标：

- 让用户在最短路径内获得 Prompt

要求：

- 支持复制主 Prompt
- 后续可扩展为复制完整 Prompt 组合
- 复制成功要有明确反馈

### 7.1.6 内容管理准备

虽然 V1 未必做完整后台，但至少要保证：

- 数据格式统一
- 内容可批量导入
- 后续能方便扩展到 CMS 或数据库

## 7.2 可选功能

以下功能可以在 V1.1 或 V2 增加：

- 登录

- 前后对比展示

## 7.3 不做功能

V1 不做：

- `id`
- `title`
- `slug`
- `model`
- `category`
- `tags`
- `prompt`

- `parameters`

- `notes`

- `source_url`

- `lang`
- `difficulty`
- `is_featured`
- `published_at`

## 10. 内容策略

## 10.1 第一批内容建议

建议优先做 5 到 10 个高价值栏目，每个栏目先做 20 到 50 条精选 Prompt。

推荐栏目：

- 产品广告图
- 人像写真
- 海报设计
- 3D 插画
- 电商主图
- 社媒封面

## 10.2 内容选择原则

第一阶段不追求绝对数量，重点在：

- 案例质量高
- 分类清晰
- 可直接复制
- 可读性强

## 10.3 内容审核标准

发布前至少检查：

- 图片是否足够清晰
- Prompt 是否可读
- 标签是否准确
- 是否重复
- 是否包含高风险内容

## 14. 技术建议

建议路线：

- 前端：`Next.js`
- 内容存储：本地 JSON / MDX 或轻量数据库
- 图片：对象存储
- 搜索筛选：基于结构化字段实现

### 14.1 当前推荐的前后端方案

考虑到本项目后续需要统计热门作品、记录用户行为、并且用户可以部署到阿里云或腾讯云服务器，推荐采用以下方案：

- 前端：`Next.js App Router`
- 后端：使用 `Next.js` 自带服务端能力与 API 路由
- 数据库：`SQLite`
- 图片资源：继续使用本地静态资源或后续迁移到对象存储
- 部署方式：优先部署到有持久磁盘的云服务器，而不是纯 Serverless 平台

这样做的优点是：

- 架构简单，开发成本低
- 前后端统一在一个项目中，便于维护
- 可以真正记录全站统计数据，而不是只做本地浏览器假数据
- 后续要扩展后台、内容管理、用户系统时，也更容易继续演进

### 14.2 为什么当前不推荐第三方统计为主方案

当前阶段不建议把“热门作品”完全依赖第三方统计服务来做主逻辑，原因包括：

- 会额外引入外部依赖
- 数据可控性弱
- 后续如果要做站内排行榜、专题推荐、精选机制，仍然需要自己的后端数据层

因此当前阶段更推荐：

`站内事件采集 + 自己的数据库统计 + 前端按统计结果排序`

而不是：

`纯第三方埋点平台 + 前端手动映射热门`

### 14.3 后端统计目标

这一版后端不是为了做复杂后台，而是为了先解决一个非常实际的问题：

- 知道哪些 Prompt 被看得最多
- 知道哪些 Prompt 被复制得最多
- 知道哪些 Prompt 被下载得最多
- 基于这些数据得到“最热门”排序

### 14.4 第一版统计事件范围

第一版建议只统计 3 类事件：

- `view`：详情页浏览
- `copy`：点击复制 Prompt
- `download`：点击下载图片

这 3 类事件已经足够支撑站内“热门度”判断，不需要一开始就做复杂埋点体系。

### 14.5 第一版数据库设计建议

建议至少有一张统计表：

#### `prompt_stats`

字段建议：

- `slug`
- `views`
- `copies`
- `downloads`
- `score`
- `updated_at`

其中：

- `slug` 作为 Prompt 的唯一关联标识
- `score` 用于热门排序
- `updated_at` 用于辅助排序和后台查看最近变化

如果后续需要更细的数据，再扩展事件日志表即可，但第一版不建议一开始就上明细日志。

### 14.6 热门度计算建议

第一版采用简单权重分即可：

`score = views * 1 + downloads * 2 + copies * 3`

这样设计的原因：

- 浏览代表兴趣
- 下载代表更强意图
- 复制代表最高复用意图

如果后续发现排序不理想，再调整权重，不需要一开始就引入复杂算法。

### 14.7 后端接口建议

建议至少提供以下接口：

#### `POST /api/prompt-stats`

用途：

- 前端在发生行为时调用
- 上报 `slug` 和事件类型
- 后端更新数据库统计

#### `GET /api/prompt-stats`

用途：

- 返回当前所有 Prompt 的热门分值
- 可同时返回每条 Prompt 的浏览、复制、下载统计
- 供前端“最热门”排序使用

### 14.8 前端接入方式建议

前端只需要在几个关键动作里触发上报：

- 进入 Prompt 详情页时，上报 `view`
- 点击复制按钮时，上报 `copy`
- 点击下载图片时，上报 `download`

浏览页的“最热门”排序逻辑则改为：

- 请求后端统计接口
- 获取每条 Prompt 的 `score`
- 在前端按分值排序

### 14.9 部署约束

如果采用 `SQLite`，推荐部署前提是：

- 使用阿里云 / 腾讯云服务器
- 服务器有持久磁盘
- 数据库文件保存在项目目录外或持久目录中

不建议把当前这套 SQLite 统计方案直接当成纯 Serverless 生产方案，因为那种环境下数据库文件不稳定。

### 14.10 第一阶段实施原则

这一版后端统计只做最小可用闭环，不做以下内容：

- 登录用户统计
- 用户级行为分析
- 复杂仪表盘
- 管理后台
- 实时图表系统
- 复杂埋点平台接入

原则是：

- 先把“热门作品”跑通
- 先让前端排序基于真实后端数据
- 先服务产品验证，而不是过度建设后台

第一阶段原则：

- 先跑通内容浏览和复制链路
- 不做过重的后台系统

## 15. 成功标准

如果 V1 达到以下状态，可以认为方向成立：

- 用户能快速理解网站价值
- 用户能在几步内找到想要的 Prompt
- 用户愿意频繁点击复制
- 内容结构支持持续扩充
- 后续收费功能有明确承接点
