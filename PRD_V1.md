# ImagePromptive PRD V1

## 1. 项目目标

本项目是一个面向 `Image 2.0` 场景的提示词内容站，核心目标不是做复杂社区或 SaaS，而是先把下面这件事做扎实：

- 让用户快速找到可直接复用的高质量提示词
- 让用户可以方便地浏览、搜索、复制、查看示例效果
- 通过轻量统计能力，识别哪些作品更受欢迎

当前阶段优先做出一个可上线、可验证、可持续补内容的版本。

## 2. 产品定位

一句话定位：

`一个专注于 Image 2.0 提示词浏览、搜索与案例展示的轻量内容站。`

当前不把项目定位为：

- 社区产品
- 登录会员平台
- 在线出图工具
- 复杂后台系统

## 3. 当前功能范围

### 3.1 对外页面

当前站点的公开结构以精简为原则，保留以下核心页面：

- `/`：首页
- `/explore`：浏览与搜索
- `/categories`：分类浏览
- `/collections`：专题推荐
- `/prompts/[slug]`：提示词详情页
- `/sources`：来源说明页

说明：

- `更新` 入口当前隐藏
- `标签` 不作为对外主导航
- `收藏` 暂不做
- `关于` 页面不做

### 3.2 V1 必做功能

- 首页展示站点价值、精选内容、分类入口、专题推荐
- 全站统一搜索入口
- 提示词列表浏览与基础排序
- 分类浏览
- 专题浏览
- 提示词详情页
- 一键复制提示词
- 基于真实数据的热门排序

### 3.3 当前不做

- 用户登录
- 收藏系统
- 评论与社区
- 支付系统
- 后台 CMS
- 站内 AI 改写 Prompt
- 复杂推荐系统

## 4. 用户核心任务

用户来到站点后，主要做这几件事：

1. 浏览灵感
2. 搜索合适的提示词
3. 查看案例图和说明
4. 复制提示词去使用

后续站点运营方还需要一件事：

5. 知道哪些内容更热门，便于调整首页、专题和内容策略

## 5. 为什么采用 Vercel + Supabase

当前阶段推荐组合：

- 前端与接口：`Next.js + Vercel`
- 数据库：`Supabase Postgres`

选择这套组合的原因：

- 部署快，适合当前 MVP 阶段
- 与现有 Next.js 项目兼容，不需要拆成前后端分离
- Supabase 免费额度适合早期项目起步
- 可以持久化真实统计数据，不再依赖手动 `popularityScore`
- 后续如果要加后台、登录、内容管理，也比纯静态方案更容易扩展

## 6. 技术方案

### 6.1 架构方式

当前不做前后端分离，采用一体化方案：

- 前端页面继续用 `Next.js App Router`
- 后端接口使用 `Next.js Route Handlers`
- 部署到 `Vercel`
- 数据库存放在 `Supabase`

这样做的优点：

- 一个仓库即可完成页面、接口和部署
- 开发成本低，维护简单
- 更适合当前内容站形态

### 6.2 为什么不继续纯前端手动热门

纯前端方案虽然能快速上线，但有明显限制：

- 无法知道真实浏览量
- 无法知道复制量
- 无法知道下载量
- “最热门”只能靠手动维护分数

因此如果要真正做“最热门”，就需要最小后端能力和数据库支持。

## 7. 第一阶段统计目标

V1 只统计最关键的 3 类行为：

- `view`：进入提示词详情页
- `copy`：点击复制提示词
- `download`：点击下载示例图

这三类数据已经足够支撑热门排序，不需要一开始就做复杂埋点系统。

## 8. 热门排序规则

V1 采用简单、可解释的评分公式：

`score = views * 1 + downloads * 2 + copies * 3`

解释：

- 浏览代表兴趣
- 下载代表更强意图
- 复制代表最高复用意图

这套权重先用于 MVP，后续可以根据实际数据再调整。

## 9. 数据库设计

### 9.1 表一：prompt_stats

用于存储每条提示词的聚合统计结果。

建议字段：

- `slug`：提示词唯一标识
- `views`：浏览次数
- `copies`：复制次数
- `downloads`：下载次数
- `score`：热门分数
- `updated_at`：最后更新时间

说明：

- 第一版先做聚合表，不急着做明细事件表
- 如果后续要看更细的行为轨迹，再补事件日志表

### 9.2 可选表二：prompt_events

这张表不是 V1 必做，但可以作为后续扩展预留。

可能字段：

- `id`
- `slug`
- `event_type`
- `created_at`
- `ip_hash`
- `user_agent`

V1 暂时不建议一开始就上线这张表，先保持轻量。

## 10. 接口设计

### 10.1 上报事件

`POST /api/prompt-stats`

用途：

- 前端发生关键行为时调用
- 上报 `slug` 和 `eventType`
- 服务端更新对应提示词的聚合统计

请求示例：

```json
{
  "slug": "minimal-product-photo",
  "eventType": "copy"
}
```

### 10.2 获取统计结果

`GET /api/prompt-stats`

用途：

- 返回当前提示词统计数据
- 支持前端“最热门”排序
- 后续也可给简单后台使用

返回字段至少包含：

- `slug`
- `views`
- `copies`
- `downloads`
- `score`

## 11. 前端接入方式

### 11.1 详情页浏览统计

用户进入 `/prompts/[slug]` 时，前端触发一次 `view` 上报。

### 11.2 复制统计

用户点击“复制提示词”按钮时，触发一次 `copy` 上报。

### 11.3 下载统计

用户点击下载示例图时，触发一次 `download` 上报。

### 11.4 浏览页热门排序

`/explore` 页的“最热门”排序不再使用手动分数，而是：

- 请求统计接口
- 获取每条提示词的 `score`
- 按 `score` 从高到低排序

## 12. 部署方案

### 12.1 前端与接口

部署平台：`Vercel`

理由：

- 对 Next.js 支持最好
- 上手快，部署流程简单
- 适合当前站点迭代节奏

### 12.2 数据库

数据库平台：`Supabase`

理由：

- 免费额度适合早期验证
- 直接提供 Postgres
- 环境变量接入简单
- 后续扩展用户系统或后台更顺滑

### 12.3 环境变量

预计需要：

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`

说明：

- `SERVICE_ROLE_KEY` 只允许服务端使用
- 前端不要直接暴露具有写权限的密钥

## 13. 风险与限制

### 13.1 当前方案的优势

- 可部署
- 可持续更新内容
- 可拿到真实热门数据
- 可继续保留当前 Next.js 结构

### 13.2 当前方案的限制

- 仍然不是完整后台系统
- 统计是轻量事件计数，不是专业分析平台
- 如果以后要做复杂运营分析，仍可能补充事件分析工具

### 13.3 面向中国大陆用户的注意点

- `Vercel + Supabase` 很适合前期快速验证
- 但如果后续主要服务中国大陆公开用户，访问稳定性和速度需要单独评估
- 如果后期流量起来，再考虑迁移到国内云厂商并不晚

## 14. 当前实施原则

这一版只做最小可用闭环：

- 真实统计
- 热门排序
- 不做复杂后台
- 不做登录
- 不做收藏
- 不做大规模架构重构

原则是：

- 先让“最热门”真正跑通
- 先让内容站具备基本数据反馈能力
- 先服务产品验证，再考虑系统扩张

## 15. 成功标准

如果 V1 达到下面这些状态，就可以认为方向成立：

- 用户能快速找到提示词
- 用户愿意频繁点击复制
- 站点可以基于真实数据识别热门内容
- 运营上可以据此调整首页、专题和内容优先级
- 技术架构足够轻，便于继续上线和维护

## 16. 下一步开发范围

确认本方案后，下一步开发按以下顺序推进：

1. 接入 `Supabase`
2. 建立 `prompt_stats` 表
3. 新增统计 API
4. 在详情页、复制按钮、下载按钮埋点
5. 把“最热门”排序改成真实后端数据驱动
6. 补一页简单的统计查看方式（可选）
